## Backend Architecture Conventions

- The Claude Agent SDK (`@anthropic-ai/claude-agent-sdk`) must only be imported in `server/src/providers/claude-adapter.ts`. All other server and UI code uses the normalized `NormalizedMessage` / `ProviderAdapter` types from `providers/types.ts`. The WebSocket protocol sends `{ type: "message" }` events with normalized payloads. The `ProviderAdapter` interface exists for testability, not multi-provider ambition — `ProviderSessionOptions` contains several Claude-specific concepts (`permissionMode`, `modeTransitionTools`, `onSubagentStarted/Completed`, `onCompacting`) with no equivalent in other SDKs.
- **SDK `systemPrompt` must use the preset** — The `sdkQuery()` call in `claude-adapter.ts` must include `systemPrompt: { type: "preset", preset: "claude_code" }`. Omitting `systemPrompt` causes the SDK to coerce `undefined` → `""` (empty string), which the process interprets as "use this empty custom prompt" — overriding the Claude Code default and producing verbose, non-tool-focused responses ("bare Claude" behavior). The SDK supports two preset-style options: `systemPrompt` and `tools` — both accept `{ type: "preset", preset: "claude_code" }`. The `tools` preset is safe to omit (defaults to the full toolset), but `systemPrompt` is not.
- **SDK `thinking` must be explicitly set** — Despite the SDK docs claiming adaptive thinking is "the default for Opus 4.6+", the runtime code only activates it when `thinking: { type: 'adaptive' }` is explicitly passed. Without it, `--max-thinking-tokens` is never sent to the CLI subprocess, and thinking never activates. The SDK translates `{ type: 'adaptive' }` → `maxThinkingTokens: 32000`.
- **SDK `effort` option** — `effort` (`'low'`/`'medium'`/`'high'`/`'max'`, default `'high'`) is passed through to `sdkQuery()` from saved settings or per-session overrides. `'max'` is Opus 4.6 only — the UI disables it for other models, and the server rejects `effort: "max"` with non-Opus models in `POST /api/sessions`.
- **SDK query options reference** — Key `sdkQuery()` options beyond what we currently use: `thinking` (`{ type: 'adaptive' }` default for Opus 4.6+, or `{ type: 'enabled', budgetTokens }` / `{ type: 'disabled' }`), `maxBudgetUsd` (hard cost cap), `additionalDirectories` (extra accessible dirs), `enableFileCheckpointing` (file change tracking/rewind), `outputFormat` (JSON Schema structured output), `betas` (e.g. `'context-1m-2025-08-07'` for 1M context on Sonnet), `agents` (programmatic subagent definitions), `mcpServers` (inline MCP config), `hooks` (SDK-level hook callbacks). Full type definitions in `server/node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts`.
- **SDK live stream sidechain filtering** — the SDK streams subagent-internal messages (prompt, tool calls, tool results) into the parent session iterator with `parent_tool_use_id != null`. These must be filtered in `normalizeMessage` before the type switch — they are never written to the parent JSONL (only to `/subagents/agent-<id>.jsonl`), so dropping them keeps the live stream consistent with history replay. The JSONL equivalent field is `isSidechain: true`.
- **`SDKSystemMessage` (`type: 'system', subtype: 'init'`)** — the very first message in every SDK stream; contains `session_id` (the real CLI session ID — same ID the CLI uses for its JSONL filename from the moment the session starts), plus `cwd`, `permissionMode`, `model`, `tools`, `skills`, and `plugins`. Useful for any data that would otherwise require waiting for the final `result` message. Handled via `onSessionIdResolved` callback in `ProviderSessionOptions`.
- **Zod schemas are the single source of truth** for all serializable API types. Define new request/response types in `server/src/schemas/`, then re-export the `z.infer<>` types from `providers/types.ts` or `types.ts`. Non-serializable types (containing functions, AbortController, Set, callbacks) stay as manual interfaces. Register new endpoints in `schemas/registry.ts` to keep the OpenAPI spec in sync.
- **API docs** are served at `/api/docs` (Scalar UI, CDN) and `/api/openapi.json` (OpenAPI 3.1 spec). Both are unauthenticated, like `/healthz`.
- **Claude adapter unit testing** — To test `claude-adapter.ts` without the real SDK, use `vi.mock("@anthropic-ai/claude-agent-sdk")` and return a mock `query()` that yields an async iterable of synthetic SDK messages. See `server/tests/claude-adapter-thinking.test.ts` for the pattern.
- **SessionWatcher is the single message authority** — All messages flow through `SessionWatcher` via `pushMessage()` (stores + broadcasts) or `pushEvent()` (broadcasts only, ephemeral). `runSession()` sets mode to `"push"` and pushes messages into the watcher; it never broadcasts directly to WebSocket clients. New subscribers default to `"poll"` mode for CLI/history live updates; `runSession()` overrides to `"push"` immediately. Do not introduce alternate delivery paths.
- **Thinking delta buffering** — `WatchedSession.pendingThinkingText` accumulates thinking text from `pushEvent()` thinking_delta events. Late-connecting WS subscribers receive the buffered text in `subscribe()`, sent before `replay_complete` to maintain correct event ordering. The buffer is cleared when: (1) `pushMessage()` receives an assistant message with a `reasoning` part, or (2) a terminal status event (completed/error/interrupted) arrives via `pushEvent()`. Assistant messages without a reasoning part do NOT clear the buffer. `subscribe()` snapshots the buffer before any `await` to prevent race conditions with concurrent `pushMessage()` calls.
- **Session ID remap lifecycle** — New API sessions start with a temp UUID. The real CLI session ID arrives in the SDK's first `init` system message (see above) — the same ID the CLI uses for its JSONL filename from the start. `runSession()` remaps early via `onSessionIdResolved` (called from `claude-adapter.ts` on `init`): updates `sessions` map, adds `sessionAliases`, calls `watcher.remap(oldId, newId)`, pushes `session_redirected`. `transitionToPoll()` still happens at session end. The late-remap fallback at session end (`result.providerSessionId !== session.sessionId`) remains for adapters like `TestAdapter` that don't call `onSessionIdResolved`. **Do not move remap back to session end** — it re-introduces the duplicate "history" session bug where `listSessionsWithHistory()` sees the live session under the temp UUID and the JSONL file under the real ID simultaneously. The client sets `currentSessionId = newId` and `connect(newId)` short-circuits (returns early, keeps the existing socket alive via `_redirectingTo`). The socket's `doConnect()` closure retains the old `sessionId` — any `sessionId`-based comparisons inside that closure are stale after remap.
